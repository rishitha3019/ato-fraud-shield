# ATO Fraud Shield

Account takeover detection system built end-to-end — from synthetic data generation through a production ML pipeline, LLM-powered explanations, REST API, and a live dashboard.

**Live demo**: [ato-fraud-shield.streamlit.app](https://ato-fraud-shield-ehtdlynk3z6gs6zkwirygy.streamlit.app)  
**API docs**: [ato-fraud-shield-api.onrender.com/docs](https://ato-fraud-shield-api.onrender.com/docs)

---

## What it does

Scores login events in real time for account takeover (ATO) fraud risk. For each login, it returns:

- A fraud probability score (0–100%)
- A risk level (HIGH / MEDIUM / LOW)
- The top contributing features via SHAP values
- A plain-English narrative generated by Claude explaining why the login was flagged and what action to take

---

## Architecture

```
Login event
    │
    ▼
FastAPI  (/predict/explain)
    │
    ├── XGBoost model  →  fraud probability + SHAP values
    │
    └── Claude (claude-haiku)  →  plain-English narrative
    │
    ▼
Streamlit dashboard  →  analyst reviews and acts
```

## Fraud patterns detected

Four distinct ATO attack types, each with different signal signatures:

- **Credential stuffing** — high failed attempts, malicious IP, new device, odd hours
- **Impossible travel** — login from a city physically unreachable given time elapsed (velocity > 900 km/h)
- **ATO session abuse** — valid credentials, new device, immediate high-velocity automated actions
- **Slow-burn ATO** — infrequent logins, subtle behavioral drift — hardest to detect

git clone https://github.com/rishitha3019/ato-fraud-shield.git
cd ato-fraud-shield
python -m venv venv && source venv/bin/activate

## Design decisions

**Why XGBoost over a neural network?** Tree-based models are faster to train, more interpretable, and better suited for tabular fraud data with engineered features. XGBoost also integrates natively with SHAP's TreeExplainer for exact (not approximate) SHAP values.

**Why SMOTE over class weighting?** With 3% fraud rate, class weighting alone left the model under-fitted on fraud patterns. SMOTE generates synthetic minority samples during training, giving the model more fraud signal to learn from without data leakage (applied to train set only).

**Why the LLM narrative layer?** SHAP values are interpretable to data scientists, but fraud operations teams need prose they can act on in seconds. The narrative layer bridges that gap — it takes the top SHAP features, translates them into domain language, identifies the attack pattern, and recommends a specific action.

**Why separate `/predict` and `/predict/explain`?** Real-time fraud scoring needs to be fast (< 100ms). The LLM call adds ~1-2 seconds. Separating the endpoints lets high-throughput systems use `/predict` while review workflows use `/predict/explain`.
